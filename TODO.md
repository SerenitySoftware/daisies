# TODO

Improvements surfaced across the May 2026 product reviews, grouped by concept.

## Reading & querying data

- [ ] **Wildcard / map traversal.** `data.users[*].email` to extract a single field across every item in a list (or every value in a dict) in one call. The single most-asked-for feature in safe-navigation libraries (cf. jq, JMESPath, Glom); without it, users still drop back to comprehensions for any non-trivial extraction. Likely the biggest expansion of the library's reach beyond single-path access.

- [ ] **Predicate queries beyond wildcards.** `chain.users.where(lambda u: u.age > 30).email` — extends the planned `users[*]` wildcard with `where()` filtering on a predicate. Closes the last gap between Daisies and JMESPath / Glom for non-trivial extraction, without forcing users to learn a string DSL. The predicate receives a `Chain` so the user gets safe navigation inside the filter too (`u.address.country == "US"` works even when `address` is missing). Pairs naturally with the wildcard work — both lower to the same internal traversal primitive. Add `.first()` / `.first_or(default)` terminals for the common "give me the first match" case.

- [ ] **Chainable `.map(fn)` for element-wise transforms.** *(PM rec, May 2026)* Transform every element of a wrapped list (or every value of a wrapped dict) without leaving the Chain: `data.users.map(lambda u: u.name.value(str).upper())` returns a Chain so the result can keep being navigated or transformed. Today the only way to transform a collection is to call `.value()` first, which drops the null-tolerance guarantee mid-pipeline — the moment the data is messy is exactly when you don't want to. The callback receives a `Chain` (safe navigation inside the mapper) and missing/empty inputs map to an empty Chain rather than raising. Lowers to the same internal traversal primitive as the planned wildcard/`where()` work and composes directly with them (`data.users.where(...).map(...)`).

- [ ] **`.fallback(...)` for multi-source default chains.** *(PM rec, May 2026)* Express "try this path, then that path, then a literal default" fluently: `data.user.email.fallback(data.contact.email).fallback("noreply@example.com")`. Real-world payloads scatter the same value across optional sources (primary vs. backup email, profile vs. account name), and today that means nesting `.value(default=...)` calls inside each other — which reads backwards and buries the intent. `.fallback()` returns the first argument that resolves to a non-missing value, accepting either another `Chain` or a plain literal, and stays Chain-wrapped so navigation can continue afterward. Small surface, very high day-to-day readability win; complements `.value()` rather than replacing it.

## Writing data

- [ ] **Mutation API with safe path creation.** `chain.user.address.city = "NYC"` writes back through the wrapped object, automatically creating any missing intermediate dicts along the way (or lists, when the path uses an integer index). `chain.user.address.pop("city")` and `del chain.user.address.city` round out the surface. Today Daisies is read-only — adding writes that respect the same null-tolerant philosophy turns it from a parse-time tool into an object-massaging tool, which is a much bigger surface. Open question to settle in the design: do mutations on a missing intermediate raise (strict) or silently create (lenient), and is that controlled by the same `STRICT_MODE` toggle as schema validation? Lean lenient by default to stay consistent with the existing read philosophy.

## Serialization & output

- [x] **`.json()` / `.dict()` / `.list()` unwrapping convenience methods.** *(PM rec, May 2026)* The most common thing a developer does after navigating a payload is send it back out — re-serialize to JSON for another API, log it, cache it, hand it to a template. Today that means either `json.dumps(chain.value())` or reaching into the private `._wrapped` attribute, both of which leak the abstraction. Add `.json(indent=None, **kwargs)` (returns a JSON string, kwargs forwarded to `json.dumps`), `.dict()` (returns the unwrapped dict, raising or returning `{}` on non-dict per the library's strictness toggle), and `.list()` (the list equivalent). High-frequency workflow gap, near-zero implementation cost, and it closes the loop so Daisies covers read → navigate → emit end-to-end rather than just the middle.

## Comparison & debugging

- [ ] **`.tree()` structural pretty-printer.** A one-call method on any Chain (`chain.tree()` or `daisies.tree(data)`) that prints the shape of the wrapped data — keys, value types, list lengths, sample primitive at each leaf, truncation for deep/wide structures. The first thing every Python developer does with an unfamiliar JSON payload is `pprint(response)`; offering a Daisies-native version that's tuned for *exploration* (types and counts, not full values) doubles as in-REPL marketing and a real productivity win for ad-hoc API debugging. Output should be terse and copy-pasteable into a bug report.

- [ ] **`chain.diff(other)` for structural comparison.** Returns a flat list of `(path, old_value, new_value, change_kind)` tuples between two `Chain` instances, where `change_kind` is one of `added | removed | changed | type_changed`. Hugely useful in tests ("did the API response shape change?"), in webhook debugging ("what's different between this delivery and the last one?"), and in migration work ("does the new schema produce the same data?"). Output is `repr`-friendly so it composes well with `pytest`'s assert-rewriting. Pairs naturally with the planned schema-validation mode — diff against the schema's "shape skeleton" to detect drift over time.

## IDE & type-safety ergonomics

- [ ] **Generate type stubs from a sample payload.** A CLI / function (`daisies stubs sample.json -o user_types.pyi`, or `daisies.stubs_from(sample)`) that walks a representative payload and emits `.pyi` stubs (or `TypedDict` / Pydantic schemas) so `chain.user.address.city` autocompletes in VS Code / PyCharm. This is the feature that flips Daisies from "convenient" to "I'm rewriting our Stripe webhook handler with this tonight" — and cleanly differentiates the library from Glom / JMESPath, which have no IDE story. Heuristics: infer Optional from sample-set nulls, infer list element types from union of observed types, name nested anonymous shapes with parent-key derivation. Pair with the planned Cookbook page to showcase real-world workflows (run `daisies stubs response.json`, get autocompletion for the rest of the file).

- [ ] **Optional schema-validation mode.** Pass a schema (a `TypedDict`, a dict-of-types, or a Pydantic model) into `Chain(data, schema=...)`; accesses that don't exist in the schema log a warning (or raise, configurable via `Chain.STRICT_MODE`). Catches "the API renamed `userId` to `user_id`" at dev time instead of via silent `None` six months later. Opt-in so it doesn't fight the library's permissive philosophy — power users who want safety get it; the default UX stays unchanged. Pairs naturally with the type-stub generator above: generate stubs once, pass the same schema in at runtime for defense in depth.
